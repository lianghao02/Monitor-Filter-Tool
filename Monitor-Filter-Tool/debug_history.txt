# 📄 卷宗四：技術攻防與除錯歷程總回顧 (debug_history.txt)

本卷宗紀載 AG-Forensic-Player 專案自開機以來的核心除錯血淚史。為確保未來在跨足不同警局與派出所環境時，技術同仁能迅速逆向檢索並排除異常，特立此除錯總表。

## 1. 開機幽靈紅字與 Big5 / UTF-8 編碼衝突
- **病因**：在微軟 Windows CMD 環境下，繁體中文系統預設採用 CP950 (Big5) 編碼。當我們的 Python 或 Batch 腳本中夾帶了 UTF-8 的 Emoji 標籤或中文提示字元時，CMD 解析器會發生字節錯位，導致如 `'01' 不是內部或外部命令` 等靈異報錯，甚至引發 Python 解譯器崩潰。
- **攻防解法**：
  - 在 `RUN.bat` 開頭宣告 `chcp 65001 >nul`，但若腳本本身存檔時帶有 UTF-16 BOM，仍會引發解析崩潰。
  - **最終大一統解法**：全面將 `RUN.bat` 洗成純淨無 BOM 的 ASCII/UTF-8 格式，移除所有干擾性 Emoji（如 `[📥]` 替換為 `[InBox]`），從底層根絕亂碼衝突。

## 2. 監視器裸流與 PyAV / HEVC 解碼崩潰
- **病因**：實戰中常常拿到 `.dav`, `.h264`, `.h265` 等缺乏標準容器封裝的「監視器裸流」，這些檔案若直接餵給 OpenCV 的 `cv2.VideoCapture`，會直接導致核心崩潰，或是無法正確定位影片時間軸。
- **攻防解法**：
  - 放棄 FFmpeg 命令列轉檔策略，全面重構導入 **PyAV (Libav 記憶體直接封裝)**。
  - 在記憶體中進行零拷貝串流解碼，成功攻克不標準的 HEVC 影像軌，並同時開發了「Timecode 自動推算與壓印技術」，將精準時間直接燒錄至截圖。

## 3. WebSocket 傳輸撐爆與 UI 無回應死鎖
- **病因**：在數位鑑識照片修復模組（AI 超解析）初期測試時，一旦遇上 4K 或是高解析度的證據截圖，透過 Eel (WebSocket) 傳輸未壓縮的 Base64 預覽圖回前端時，會直接撐爆 WebSocket 的 payload 上限，導致網頁瞬間碎屏、後台 Python 無聲卡死。
- **攻防解法**：
  - 在後台將準備送回前端「預覽用」的照片，強制使用 `cv2.imencode('.jpg', img, [cv2.IMWRITE_JPEG_QUALITY, 90])` 進行極限瘦身。
  - 此舉將封包體積瞬間縮減了 80%，實現了 0 秒秒傳，並加裝了 15 秒超時自動斷尾求生鎖，澈底消滅了卡死盲區。（實際存檔時仍採用 Quality=100 的無損陣列寫出）。

## 4. opencv-contrib 內傷與 dnn 模組遺失
- **病因**：部分電腦預設安裝了基礎版的 `opencv-python`，缺乏深度學習 `cv2.dnn_superresolution` 擴充模組。導致一啟動修復模態就拋出致命錯誤 `AttributeError: module 'cv2' has no attribute 'dnn_superresolution'`。
- **攻防解法**：
  - **前線環境防禦**：升級 `RUN.bat` 一鍵啟動器，主動偵測並強制執行 `pip uninstall opencv-python -y` 後替換為 `opencv-contrib-python`。
  - **後方演算法防禦**：在 Python 內部加裝雙軌自適應防禦鎖 `hasattr(cv2, 'dnn_superresolution')`。若發現在無網路的派出所環境且缺乏 contrib 模組，系統立刻發動「第二軌備援：Lanczos-4 內插放大 + CLAHE 直方圖均衡化」，確保照片修復功能擁有不壞之身、滿血運行。
